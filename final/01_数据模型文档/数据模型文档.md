## 二、业务背景

### 2.1 业务模型

积分兑换平台是一个面向**多商家、跨品牌**的消费者积分通兑平台：

```
消费者
  ↓ 购买带积分标识的商品
  ↓ 商家发放积分码
消费者
  ↓ 上传积分码到平台
平台
  ↓ 校验、累加积分
消费者账户
  ↓ 用积分兑换礼品
  ↓ 生成订单
  ↓ 礼品配送
完成
```

### 2.2 三大主数据

| 主数据 | 数量级 | 说明 |
|--------|--------|------|
| 商家 | 13 | 联盟商家（如圣元、可口可乐、伊利）|
| 会员 | 5 万 | 平台注册会员 |
| 礼品 | 16 | 平台可兑换礼品 |

### 2.3 三大业务凭证

| 凭证 | 数量 | 用途 |
|------|------|------|
| 积分码 (JFCode) | 4.6 万 | 商家发放 → 消费者上传 → 平台积分 |
| 订单 (OrderInfo) | 532 | 消费者兑换礼品 |
| 交易日志 (AccountTradeLog) | 7.8 万 | 积分变动流水（借贷记账）|

### 2.4 关键业务规则

#### 借贷记账
```
会员获得积分 = 商家积负债（负分）
会员兑换积分 = 商家负债减少
总积分为 0（永远平衡）
```

#### 三态积分
```
可用余额 ValidCoin    →   冻结 FrozenCoin  →   在途 OnWayCoin
     ↑                    兑换时立即冻结         配送完成时扣除
     └─ 退款时回退 ───────┘                └── 取消时退回余额
```

#### 先进先出
```
会员有多商家积分时，兑换优先消耗最早获得的商家积分
```

## 四、数据仓库模型（DW 层）

### 4.1 设计原则

按 **Kimball 维度建模** 范式：

```
★  1 事实表 + N 维表（星型模型）
    ├── 事实表：度量业务事件（可加性指标）
    └── 维表：分析角度（时间/商家/会员...）
```

**命名规范**：
- 事实表：`Fact_<业务主题>`
- 维表：`Dim_<维度名>`
- 代理键：`<表名>Key`（int IDENTITY）
- 业务键：保留源库字段名（如 `BusinessID`）

#### Dim_Hour（小时维，代码生成）

```sql
CREATE TABLE dbo.Dim_Hour (
    HourKey    TINYINT      NOT NULL PRIMARY KEY,  -- 0-23
    HourName   NVARCHAR(20) NULL,                 -- '08:00'
    TimeBucket NVARCHAR(20) NULL                  -- '早高峰/午高峰/晚高峰/其他'
);
INSERT INTO dbo.Dim_Hour (HourKey, HourName, TimeBucket)
SELECT h, RIGHT('0' + CAST(h AS VARCHAR(2)), 2) + ':00',
       CASE
         WHEN h BETWEEN 7  AND 9  THEN N'早高峰'
         WHEN h BETWEEN 11 AND 13 THEN N'午高峰'
         WHEN h BETWEEN 17 AND 21 THEN N'晚高峰'
         ELSE N'其他时段'
       END
FROM (SELECT TOP 24 ROW_NUMBER() OVER (ORDER BY (SELECT NULL)) - 1 AS h
      FROM sys.all_objects) x;
```

### 4.5 BI 视图（3 个）

```sql
-- 会员账户余额
CREATE VIEW dbo.v_Member_Account AS
SELECT c.CustomerID, c.RealName, c.FromBusiness, bm.BusinessCnName,
       a.ValidCoin, a.FrozenCoin, a.OnWayCoin, a.HistoryCoin, a.AccStatus
FROM dbo.Account a
JOIN dbo.CustomerInfo c ON a.OwnerID = c.CustomerID
LEFT JOIN dbo.BusinessMen bm ON c.FromBusiness = bm.BusinessID
WHERE a.Acctype = 1;

-- 商家负债账户
CREATE VIEW dbo.v_Merchant_Account AS
SELECT bm.BusinessID, bm.BusinessCnName,
       -a.ValidCoin AS TotalDebt,        -- 商家已发放积分总额
       a.HistoryCoin, a.AccStatus
FROM dbo.Account a
JOIN dbo.BusinessMen bm ON a.OwnerID = bm.BusinessID
WHERE a.Acctype = 2;

-- 仪表盘 KPI 综合
CREATE VIEW dbo.v_Dashboard_KPI AS
SELECT
  (SELECT COUNT(*) FROM dbo.BusinessMen WHERE BusinessStatus=1) AS MerchantCount,
  (SELECT COUNT(*) FROM dbo.CustomerInfo WHERE CusStatus=1) AS MemberCount,
  (SELECT COUNT(*) FROM dbo.GiftInfo WHERE GiftStatus=1) AS GiftCount,
  (SELECT ISNULL(SUM(TotalCoin),0) FROM dbo.OrderInfo WHERE OrderStatus IN (1,2,3)) AS TotalCoin,
  (SELECT COUNT(*) FROM dbo.OrderInfo WHERE OrderStatus IN (1,2,3)) AS OrderCount,
  (SELECT ISNULL(SUM(ValidCoin),0) FROM dbo.Account WHERE Acctype=1) AS EarnCoin;
```

## 六、模型设计要点

### 6.1 代理键 vs 业务键

| 键 | 来源 | 用途 |
|----|------|------|
| **代理键**（`MerchantKey` 等）| IDENTITY 自增 | DW 内部主键，**用于事实表外键** |
| **业务键**（`BusinessID` 等）| 源库原值 | 用于关联源数据、回溯 |

**原则**：事实表**只引用代理键**，不用业务键。

### 6.2 DateKey 设计

```sql
-- 2026-07-06 → DateKey = 20260706
DateKey = YEAR(@date) * 10000 + MONTH(@date) * 100 + DAY(@date)
```

**优势**：
- int 类型，4 字节，比 datetime 节省
- 自然排序等于时间顺序
- 分区友好（按月分区：202607*）

### 6.3 缓慢变化维（SCD）

| 类型 | 实现 | 适用 |
|------|------|------|
| Type 1（覆盖）| MERGE 直接 UPDATE | 商家名称变更（不重要）|
| Type 2（新增行）| 加 EffectiveDate/ExpiryDate | 会员等级变更（要追溯）|

本项目使用 **Type 1**。生产环境升级建议：
```sql
ALTER TABLE dbo.Dim_Merchant ADD EffectiveDate DATETIME, ExpiryDate DATETIME;
```

### 6.4 预聚合表

`Fact_Order_Daily` 是 ETL 后的**预聚合宽表**，用于加速仪表盘 30 天趋势查询：
- 输入：50 万行 OrderInfo
- 输出：89 行（30 天 × 多维度）
- 加速比：~5000 倍

### 6.5 命名约定

| 对象 | 约定 | 示例 |
|------|------|------|
| 事实表 | `Fact_` + 业务主题 | Fact_Point_Earn |
| 维表 | `Dim_` + 维度名 | Dim_Date |
| 代理键 | `<表名>Key` | MerchantKey |
| 日期键 | `DateKey` | 20260706 |
| 退化维度 | 业务键直接放事实表 | OrderID |
